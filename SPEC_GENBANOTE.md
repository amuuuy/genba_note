## 1. プロジェクト概要

### 1.1 アプリケーション情報

| 項目 | 内容 |
|------|------|
| **アプリ名** | GenBa Note（現場ノート） |
| **バンドルID** | com.kenjimu.genbanote |
| **プラットフォーム** | iOS / Android（Expo） |
| **言語** | 日本語 |

### 1.2 アプリ概要

GenBa Note（現場ノート）は、建設業・施工業者向けの見積書・請求書管理モバイルアプリケーションです。現場での迅速な見積作成から請求書発行、売上管理までをワンストップで実現します。

### 1.3 ターゲットユーザー

| ユーザー層 | 特徴 |
|-----------|------|
| **主要ターゲット** | 個人事業主の施工業者、小規模建設会社 |
| **想定規模** | 1〜10名程度の事業者 |
| **ITリテラシー** | スマートフォン基本操作が可能なレベル |
| **利用シーン** | 現場での見積作成、事務所での請求書発行 |

### 1.4 ビジネス目標

1. **効率化**: 手書き見積書からのデジタル化による作業効率向上
2. **正確性**: 自動計算による計算ミスの防止
3. **追跡性**: 見積〜請求〜入金のステータス管理
4. **収益化**: Pro機能によるサブスクリプションモデル

## 2. 機能要件

### 2.1 書類管理（見積書・請求書）

#### 2.1.1 概要

見積書と請求書の作成・編集・削除・一覧表示機能を提供します。

**書類削除時の処理:**

| 操作 | 処理内容 |
|------|---------|
| 削除確認 | 削除前に確認ダイアログを表示 |
| 書類データ削除 | AsyncStorageから書類データを削除 |
| 機微情報スナップショット削除 | expo-secure-storeから `issuer_snapshot_{documentId}` を削除 |
| 書類番号 | 削除された書類の番号は欠番として再利用しない（2.1.5参照） |

> **注記**: 書類削除時は、書類データ（AsyncStorage）と機微情報スナップショット（expo-secure-store）の両方を必ず削除する。これにより、削除後に機微情報が端末内に残存することを防ぐ。

#### 2.1.2 書類タイプ

| タイプ | 識別子 | 説明 |
|--------|--------|------|
| 見積書 | `estimate` | 工事の見積金額を提示する書類 |
| 請求書 | `invoice` | 工事完了後の請求金額を提示する書類 |

#### 2.1.3 書類ステータス

| ステータス | 識別子 | 説明 | 対象書類 |
|-----------|--------|------|----------|
| 下書き | `draft` | 作成中・未送付の状態 | 見積書・請求書 |
| 送付済 | `sent` | 取引先に送付済みの状態 | 見積書・請求書 |
| 入金済 | `paid` | 入金が確認された状態 | 請求書のみ |

##### 2.1.3.1 ステータス遷移ルール

**許可される遷移:**

```
見積書:  draft ⇔ sent
請求書:  draft ⇔ sent ⇔ paid
```

| 現在 → 次 | 許可 | 条件 |
|-----------|------|------|
| draft → sent | ◯ | なし |
| sent → draft | ◯ | なし（修正のため差し戻し） |
| sent → paid | ◯ | 請求書のみ。paidAt入力必須 |
| paid → sent | ◯ | 請求書のみ。paidAtはnullにリセット |
| paid → draft | ◯ | 請求書のみ。paidAtはnullにリセット |
| draft → paid | ✕ | sentを経由する必要あり |

> **設計意図**: 双方向の遷移を許可することで、誤操作の修正や請求書の再発行に対応します。ただし `draft → paid` は業務フロー上不自然なため禁止し、必ず `draft → sent → paid` の順序を経由させます。

##### 2.1.3.2 編集可否ルール

| フィールド | draft | sent | paid | 備考 |
|-----------|-------|------|------|------|
| 取引先名 | ◯ | ◯ | ✕ | 入金済は確定書類のため編集禁止 |
| 取引先住所 | ◯ | ◯ | ✕ | 同上 |
| 件名 | ◯ | ◯ | ✕ | 同上 |
| 発行日 | ◯ | ◯ | ✕ | 同上 |
| 有効期限 | ◯ | ◯ | - | 見積書のみ |
| 支払期限 | ◯ | ◯ | ✕ | 請求書のみ |
| 入金日 | - | - | ◯ | paid時のみ編集可（日付修正用） |
| 備考 | ◯ | ◯ | ✕ | 同上 |
| 明細行 | ◯ | ◯ | ✕ | 追加・編集・削除すべて |
| ステータス | ◯ | ◯ | ◯ | 遷移ルールに従う |

> **paid状態の編集制限**: 入金済の書類は会計処理上確定したものとして扱うため、金額に影響するフィールド（明細行、取引先情報等）の編集を禁止します。入金日のみ修正可能とし、誤入力時の訂正に対応します。

> **sent状態の編集**: 送付済であっても、実務上は修正版を再送することがあるため、全フィールドの編集を許可します。ただし、UIで「送付済の書類を編集しますか？」の確認ダイアログを表示することを推奨します。

> **編集による不変条件の維持**: フィールド編集時、日付の整合条件（2.1.4参照）およびpaidAtの制約を常に検証し、違反する場合はエラーを表示して保存を拒否します。

#### 2.1.4 書類基本情報

| フィールド | 必須 | 説明 | 見積書 | 請求書 |
|-----------|------|------|--------|--------|
| 書類番号 | ◯ | 自動採番（例: EST-001, INV-001） | ◯ | ◯ |
| 取引先名 | ◯ | 顧客・会社名（書類に埋め込み保存） | ◯ | ◯ |
| 取引先住所 | - | 顧客の住所（書類に埋め込み保存） | ◯ | ◯ |
| 件名 | - | 工事名・案件名 | ◯ | ◯ |
| 発行日 | ◯ | 書類の発行日（YYYY-MM-DD）。未来日も許容 | ◯ | ◯ |
| 有効期限 | - | 見積の有効期限。issueDate以降の日付のみ | ◯ | - |
| 支払期限 | - | 請求書の支払期限。issueDate以降の日付のみ | - | ◯ |
| 入金日 | 条件付 | 入金確認日（YYYY-MM-DD）。status='paid'時は必須 | - | ◯ |
| 備考 | - | 自由記述欄 | ◯ | ◯ |

> **取引先情報のデータモデル**: 取引先情報は書類ごとに埋め込み保存します（マスタ参照ではない）。これにより、取引先名の変更が過去の書類に影響しません。

> **日付の整合条件**:
> - 有効期限（validUntil）≥ 発行日（issueDate）
> - 支払期限（dueDate）≥ 発行日（issueDate）
> - 発行日は未来日も許容（予定作成のユースケース対応）

> **日付の保存・比較・パース方針**:
> - **保存形式**: すべての日付は `YYYY-MM-DD` 形式の**文字列**として保存（ISO 8601のdate部分のみ）
> - **未設定値の表現**: 任意項目（validUntil, dueDate, paidAt）の未設定は **`null`** で統一。空文字やキー省略は使用しない
> - **パース禁止**: `new Date('YYYY-MM-DD')` はUTC解釈されるため**使用禁止**。タイムゾーン境界で1日ずれるバグの原因となる
> - **パース方法**: 日付文字列を年/月/日に分割し、明示的にローカル日付として扱う
>   - 例: `const [y, m, d] = '2026-01-30'.split('-').map(Number); const localDate = new Date(y, m - 1, d);`
> - **比較方法**: 文字列比較（`'2026-01-30' >= '2026-01-01'`）または上記でパースしたローカル日付同士で比較。**nullの場合は比較対象から除外**
> - **「今日」の取得**: `new Date()` からローカルの年/月/日を取得し、`YYYY-MM-DD` 文字列に変換して使用
> - **CSV出力時の変換**: null → 空文字（`""`）として出力（2.6.6参照）
> - **実装ガイドライン**: 日付操作用のユーティリティ関数を作成し、アプリ全体で統一して使用すること

> **入金日とステータスの整合ルール（不変条件）**:
> - `status = 'paid' ⇔ paidAt が設定済み`（双方向制約）
> - status を 'paid' に変更する際は入金日（paidAt）の入力が必須
> - status が 'draft' または 'sent' の場合、paidAt は常に null
> - paidAt を削除する場合は status を 'draft' または 'sent' に戻す必要あり
> - paidAt は issueDate 以降 かつ **今日以前**の日付のみ許容（未来日は禁止）
> - **「今日」の判定基準**: 端末ローカルタイムゾーンの暦日（00:00:00〜23:59:59）。収支集計の期間フィルタ（2.6.2参照）と同一基準
> - 「入金確認日」として、実際に入金を確認した日付を記録する
>
> **未来発行日の請求書について**:
> - issueDateが未来の場合、paidAtの制約（issueDate以降かつ今日以前）を満たせないため、status='paid'にはできない
> - UIでは、未来発行日の請求書に対してステータス変更ボタンを「下書き」「送付済」のみに制限し、「入金済」は選択不可とする
> - 発行日が今日以前になった時点で「入金済」への変更が可能になる
> - **sent ステータスは許可**: 未来発行日でも「送付済」への変更は可能（予約送信・事前準備のユースケース）
> - **収支集計での扱い**: 未来発行日の請求書も issueDate が選択期間内であれば集計・CSV出力対象に含まれる（発行日基準で計上するため）

#### 2.1.5 自動採番ルール

| 項目 | 仕様 |
|------|------|
| **見積書プレフィックス** | `EST-`（設定で変更可能） |
| **請求書プレフィックス** | `INV-`（設定で変更可能） |
| **番号桁数** | 3桁ゼロパディング（001〜999） |
| **上限到達時** | 999 → 1000 と連番継続（桁数は自動拡張） |
| **採番タイミング** | 書類を初回保存した時点（下書き保存含む） |
| **欠番許容** | 許容する。削除・キャンセルされた番号は再利用しない |
| **永続化** | AsyncStorageで番号状態を保持 |
| **プレフィックス変更時** | 連番は継続（リセットしない）。変更後の新規採番にのみ反映 |
| **編集可否** | 書類番号は個別編集不可（ユーザーは既存書類の番号を変更できない） |
| **プレフィックス制限** | 半角英数字・ハイフン・アンダースコアのみ（1〜10文字） |
| **ユニーク性スコープ** | 書類タイプ別（見積書と請求書で独立した連番） |

> **採番ルールの補足**:
> - 番号は一度発行したら再利用しない（欠番が生じてもカウンタは進める）
> - 書類作成を開始しても、保存せずにキャンセルした場合は番号を消費しない
> - 書類を削除しても、その番号は欠番となり再利用されない
> - 書類番号はシステムが自動で付与し、ユーザーが任意に変更することはできない
> - 見積書と請求書はそれぞれ独立した連番を持ち、同一番号でも書類タイプが異なれば衝突しない（例: EST-001 と INV-001 は別の書類として存在可能）

**例:**
- 見積書: EST-001, EST-002, ..., EST-999, EST-1000, EST-1001...
- 請求書: INV-001, INV-002, ..., INV-999, INV-1000, INV-1001...

#### 2.1.6 見積書 → 請求書変換

見積書を元に請求書を新規作成する機能です。

| 項目 | 変換時の動作 |
|------|-------------|
| 書類ID | 新規生成（UUID） |
| 書類番号 | **変換実行時に自動保存し、請求書として新規採番を確定** |
| タイプ | `estimate` → `invoice` |
| 発行日 | 変換実行日 |
| 有効期限 | 削除 |
| 支払期限 | 空（手動入力） |
| ステータス | `draft` |
| 取引先情報 | 見積書からコピー |
| 明細行 | 見積書からコピー |
| 備考 | 見積書からコピー |
| 発行者情報スナップショット | **変換時点の設定から再取得**（見積書のコピーではない。2.3.2参照） |
| 機微情報スナップショット | **変換時点の設定から再取得**（expo-secure-storeに新規保存） |

> **採番タイミングの明確化**: 見積→請求変換時は、変換実行と同時に請求書を自動保存し、書類番号を確定します。これにより、変換後に編集画面でキャンセルしても書類番号は消費済みとなります（2.1.5の「初回保存時に採番」ルールに準拠）。ユーザーは変換実行前の確認ダイアログで番号消費について認識できます。

### 2.2 取引先情報

> **注記**: 取引先情報は書類に直接埋め込んで保存します。取引先マスタは持たず、書類作成時に都度入力します。これにより、取引先名の変更が過去の書類に影響しません。

#### 2.2.1 取引先フィールド

| フィールド | 必須 | 説明 |
|-----------|------|------|
| 取引先名 | ◯ | 顧客・会社名（書類に埋め込み保存） |
| 取引先住所 | - | 顧客の住所（書類に埋め込み保存） |

### 2.3 発行者情報（自社情報）

#### 2.3.1 発行者フィールド

| フィールド | 必須 | 説明 |
|-----------|------|------|
| 会社名 | - | 自社名・屋号 |
| 代表者名 | - | 代表者氏名 |
| 住所 | - | 自社住所 |
| 電話番号 | - | 連絡先電話番号 |
| インボイス番号 | - | 適格請求書発行事業者登録番号 |

> **注記**: 発行者情報はすべて任意項目です。書類保存・PDF出力は発行者情報未設定でも可能です。PDF上では未設定項目は非表示になります。

#### 2.3.2 発行者情報・振込先情報のスナップショット方針

| 項目 | 方針 |
|------|------|
| **スナップショットタイミング** | 書類を**初回保存**した時点で、設定画面の現在値をスナップショットとして保存 |
| **設定変更の影響** | 設定変更は既存書類に影響しない。新規作成する書類にのみ反映 |
| **paid書類の不変性** | 入金済書類のスナップショットは編集不可（2.1.3.2の編集可否ルールに準拠） |
| **draft/sent書類の更新** | 「設定から再読み込み」操作でスナップショットを最新の設定値で上書き可能（UI確認あり） |

**スナップショットの保存先と参照方式:**

| 情報分類 | 対象フィールド | 保存先 | 参照方式 |
|---------|---------------|--------|---------|
| **一般発行者情報** | 会社名、代表者名、住所、電話番号 | 書類データ内に埋め込み（AsyncStorage） | 書類から直接参照 |
| **機微情報** | インボイス番号、振込先口座情報（銀行名、支店名、口座種別、口座番号、口座名義） | expo-secure-store（書類IDをキーとして関連付け） | **HTMLプレビュー表示時およびPDF生成時**にsecure-storeから取得して結合 |

> **設計意図**:
> - 一般発行者情報（会社名等）は書類データに直接埋め込み、取引先情報（2.2参照）と同じ方式でスナップショット保存する
> - 機微情報（インボイス番号・振込先口座）は3.2のセキュリティ方針に従いexpo-secure-storeに保存し、暗号化で保護する
> - 機微情報は書類IDをキーとして `issuer_snapshot_{documentId}` の形式でsecure-storeに保存
> - **HTMLプレビュー表示時およびPDF生成時**は、書類データ（AsyncStorage）と機微情報スナップショット（secure-store）を結合して出力
> - **CSVエクスポートには機微情報を含めない**（金額集計用途のため、振込先口座・インボイス番号は不要。詳細は2.6.6参照）
> - これにより、設定変更が過去の書類出力に影響しない不変性と、機微情報の暗号化保護を両立する

### 2.4 明細行管理

#### 2.4.1 明細行フィールド

| フィールド | 必須 | 型 | 説明 |
|-----------|------|-----|------|
| 品名 | ◯ | string | 工事項目・作業内容 |
| 数量 | ◯ | number | 数量（0.001以上、小数点以下3桁まで） |
| 単位 | ◯ | string | 単位（式、個、m、m²、人工、台等） |
| 単価 | ◯ | integer | 円単位の単価（0以上） |
| 税率 | ◯ | 0 \| 10 | 消費税率（0%または10%）※建設業は軽減税率(8%)対象外のため |
| 小計 | 自動 | integer | 数量 × 単価（自動計算、円未満切り捨て） |

#### 2.4.2 明細行操作

| 操作 | 説明 |
|------|------|
| 追加 | 新規明細行を末尾に追加 |
| 編集 | 既存明細行の内容を変更 |
| 削除 | 明細行を削除（確認ダイアログあり） |
| 単価表連携 | 単価表から品名・単価・税率を自動入力 |

#### 2.4.3 金額計算ロジック

**設計原則:**
- 全ての金額は**整数（円）**で保持（浮動小数点の丸め誤差を回避）
- 数量は小数点以下3桁まで許容（内部では1000倍した整数で保持: quantityMilli）
- 消費税は**切り捨て**（Math.floor）

**計算式（整数演算）:**

```
// 実装は整数演算を使用し、浮動小数点演算は使用しない
明細行小計 = Math.floor(quantityMilli * unitPrice / 1000)  // 円未満切り捨て

税抜合計（subtotalYen） = Σ 明細行小計

消費税（taxYen） = Σ Math.floor(明細行小計 × 税率 / 100)

税込合計（totalYen） = 税抜合計 + 消費税
```

**税率別内訳の計算規則:**
- 消費税は明細行単位で計算・切り捨てを行い、税率ごとに合算する
- PDF/CSV/UIで表示する税率別内訳は、この計算結果を使用する
- 例: 10%対象の明細が3行ある場合、各行の税額を切り捨て後に合算
- これにより、UI/PDF/CSVで一貫した金額が表示される

**数量の内部表現:**
- 入力: 小数（例: 2.5）、小数点以下3桁まで許容
- 内部保持: 1000倍した整数（例: 2500）= quantityMilli
- 計算時: `lineSubtotal = Math.floor(quantityMilli * unitPrice / 1000)`
- 入力制限: 3桁を超える小数は入力時に3桁で切り捨て（四捨五入ではない）

**金額計算の上限値（安全整数範囲の保証）:**

JS Numberの安全整数範囲（`Number.MAX_SAFE_INTEGER` = 9,007,199,254,740,991）を超えないよう、以下の制約を設定:

**静的上限値:**

| 項目 | 上限値 | 備考 |
|------|--------|------|
| 数量（quantity） | 99,999.999 | quantityMilli = 99,999,999（実用的な最大数量） |
| 単価（unitPrice） | 99,999,999円 | 約1億円（大規模工事に対応） |
| 明細行数 | 100行 | 1書類あたり |
| 書類の税込合計 | 9,999,999,999円 | 約100億円（明細小計の合計上限） |

**動的制約（必須）:**

```
quantityMilli × unitPrice ≤ Number.MAX_SAFE_INTEGER（約9×10^15）
```

> **検証タイミングと方針**:
>
> | 上限 | 検証タイミング | 検証方法 |
> |------|--------------|---------|
> | 数量・単価の静的上限 | 数量・単価の入力確定時 | 入力値が上限を超える場合はエラー表示して入力を拒否 |
> | 動的制約（quantityMilli × unitPrice） | 数量・単価の入力確定時 | `quantityMilli * unitPrice > MAX_SAFE_INTEGER` の場合はエラー表示して入力を拒否 |
> | 明細行数上限（100行） | 明細追加操作時 | 100行に達している場合は「追加」ボタンを無効化し、追加を拒否 |
> | 税込合計上限（9,999,999,999円） | 明細変更・保存時 | 税込合計が上限を超える場合はエラー表示して保存を拒否 |
>
> 動的制約により、数量と単価の組み合わせがMAX_SAFE_INTEGERを超えないことを保証する。例: 数量10,000（quantityMilli=10,000,000）の場合、単価上限は約9億円に制限される。

> **上限設計の根拠**: 静的上限の最大組み合わせ `quantityMilli(99,999,999) × unitPrice(99,999,999)` = 約 9.99 × 10^15 で、`Number.MAX_SAFE_INTEGER`（約 9.0 × 10^15）を超える。**したがって、静的上限だけでは安全整数範囲を保証できない。動的制約チェックが常に優先され必須である。** 中小建設業の実務では数量1,000未満 × 単価1,000万未満の組み合わせが一般的であり、動的制約に抵触することは稀だが、入力時に必ず検証を行う

**例:**
| 品名 | 数量 | 単価 | 税率 | 小計 | 消費税 |
|------|------|------|------|------|--------|
| 外壁塗装 | 1 | 150,000 | 10% | 150,000 | 15,000 |
| 足場代 | 1 | 30,000 | 10% | 30,000 | 3,000 |
| **合計** | | | | **180,000** | **18,000** |
| **税込合計** | | | | | **198,000** |

### 2.6 収支管理

#### 2.6.1 概要

請求書データを元に売上を集計・分析する機能です。

#### 2.6.2 集計機能

| 機能 | 説明 |
|------|------|
| 期間フィルタ | 今月 / 過去3ヶ月 / 今年 |
| 期間合計 | 選択期間の総売上・入金済/未入金内訳 |
| 月別集計 | 月ごとの売上・入金状況 |
| 請求書詳細 | 月別の請求書一覧表示 |

**期間フィルタの境界定義:**
- 基準タイムゾーン: 端末ローカルタイムゾーン（アプリ全体で統一）
- 今月: 当月1日 00:00:00 〜 当月末日 23:59:59（端末ローカルTZ）
- 過去3ヶ月: 当月含む直近3ヶ月（例: 1月なら11月〜1月）の月初〜月末
- 今年: 当年1月1日 00:00:00 〜 当年12月31日 23:59:59（端末ローカルTZ）

> **タイムゾーン統一方針**: 収支集計、CSVエクスポート、その他すべての期間フィルタで端末ローカルタイムゾーンを基準とします。海外渡航時も端末設定に従います。

#### 2.6.3 集計基準日

全ての集計項目は**発行日（issueDate）基準**で統一し、「合計金額 = 入金済金額 + 未入金金額」が常に成立するようにします。

| 集計項目 | 基準日 | 説明 |
|---------|--------|------|
| 期間フィルタ | 発行日（issueDate） | 請求書の発行日が期間内かどうかで抽出 |
| 売上合計 | 発行日（issueDate） | 期間内発行 かつ status in [sent, paid] の請求書総額 |
| 入金済金額 | 発行日（issueDate） | 期間内発行 かつ status='paid' の請求書合計 |
| 未入金金額 | 発行日（issueDate） | 期間内発行 かつ status='sent' の請求書合計 |

> **設計方針**:
> - 売上として計上するのは送付済（sent）および入金済（paid）のみ
> - 下書き（draft）は売上に含めない（作成中のため）
> - 入金日（paidAt）ではなく発行日（issueDate）基準で集計することで、同一期間の内訳合計が一致します
> - **未来発行日の請求書**: issueDateが選択期間内であれば集計対象に含まれる（発行日基準で計上するため）。ただし未来発行日の請求書はpaid不可（2.1.4参照）のため、sentとして未入金金額に計上される

#### 2.6.4 表示項目

| 項目 | 説明 |
|------|------|
| 売上合計 | 期間内発行 かつ status in [sent, paid] の請求総額（税込） |
| 入金済金額 | issueDate が期間内 かつ status='paid' の請求書合計 |
| 未入金金額 | issueDate が期間内 かつ status='sent' の請求書合計 |
| 請求書件数 | 期間内の請求書数（issueDate 基準、sent/paid のみ） |

#### 2.6.5 CSVエクスポート【Pro限定】

請求書データをCSV形式で出力する機能です。

| 項目 | 説明 |
|------|------|
| 対象 | **issueDate が選択期間内** かつ **status in [sent, paid]** の請求書（draft は除外） |
| 基準日 | issueDate（収支集計と同一基準） |
| 期間境界 | 選択期間の開始日00:00:00〜終了日23:59:59（端末ローカルTZ、2.6.2と同一） |
| 形式 | CSV（UTF-8 BOM付き） |
| 改行コード | CRLF |
| ファイル名 | `invoices_YYYYMMDD.csv`（エクスポート実行日） |
| 制限 | **Proプラン限定機能** |

> **収支集計との整合性**: CSV出力は収支集計画面と同じ抽出条件（issueDate基準、sent/paidのみ）を使用するため、エクスポートデータと画面表示の合計値が一致します。

#### 2.6.6 CSVカラム定義

| # | カラム名 | 説明 | フォーマット |
|---|---------|------|-------------|
| 1 | documentNo | 書類番号 | 文字列（例: INV-001） |
| 2 | issueDate | 発行日 | YYYY-MM-DD |
| 3 | dueDate | 支払期限 | YYYY-MM-DD（未設定時は空） |
| 4 | paidAt | 入金日 | YYYY-MM-DD（未設定時は空） |
| 5 | clientName | 取引先名 | 文字列 |
| 6 | subject | 件名 | 文字列（未設定時は空） |
| 7 | subtotalYen | 税抜合計 | 整数（円） |
| 8 | taxYen | 消費税 | 整数（円） |
| 9 | totalYen | 税込合計 | 整数（円） |
| 10 | status | ステータス | sent/paid（draft は出力対象外のため含まれない） |

> **エスケープ規則**: カンマ・ダブルクォート・改行を含むフィールドはダブルクォートで囲む。ダブルクォート自体は二重化（""）。

> **status カラムについて**: CSV出力は `status in [sent, paid]` のみを対象としているため、出力されるCSVファイルには draft ステータスのレコードは含まれません。

**オフライン時のPro判定方針:**

| 項目 | 仕様 |
|------|------|
| 判定データソース | **expo-secure-storeに保存された4値**（`entitlement_active`、`entitlement_expiration`、`last_verified_at`、`last_verified_uptime`）を使用 |
| RevenueCatキャッシュ | オンライン検証時のデータ取得に使用。**オフライン判定には直接使用しない**（改ざん耐性のため） |
| オフライン時の判定 | **下記「オフラインPro許可の最終判定式」に従う**（簡易判定は禁止） |
| 再検証タイミング | オンライン復帰時にRevenueCat SDKでサーバーと同期し、成功時にsecure-storeの4値を更新 |

**グレース期間（オフライン継続許可）:**

| 項目 | 仕様 |
|------|------|
| グレース期間上限 | **7日間**（最終オンライン検証日から起算） |
| 有効条件 | **下記「オフラインPro許可の最終判定式」の4条件すべてを満たす場合のみPro機能を許可** |
| 失効条件 | 4条件のいずれか1つでも満たさない場合、Pro機能を無効化し無料プランにフォールバック |
| 失効時の挙動 | オンライン接続を促すメッセージを表示。接続後に再検証 |

> **オフラインファーストとの整合性**: オフライン環境でもPro機能を利用可能とするが、課金保護のため4条件すべてを満たす必要がある。ネットワーク接続は購入処理・復元処理・検証処理に必要だが、条件を満たせば日常的なPro機能利用はオフラインで完結する

> **判定データのソース・オブ・トゥルース**: オフライン判定は **expo-secure-storeに保存された4値**（`entitlement_active`、`entitlement_expiration`、`last_verified_at`、`last_verified_uptime`）をソース・オブ・トゥルースとする。RevenueCat SDKのローカルキャッシュは参考情報として扱い、オフライン判定には使用しない（改ざん耐性のため）

> **グレース期間7日の根拠**: 現場作業者が1週間程度オフライン環境で作業するケースを想定。7日を超える長期オフラインは稀であり、課金保護とユーザー利便性のバランスを考慮した設定

**時刻改ざん対策（uptimeベース）:**

| 項目 | 仕様 |
|------|------|
| 保存データ | expo-secure-storeに以下を保存: |
| | - `entitlement_active`: エンタイトルメントがアクティブか（`entitlement.isActive == true`、boolean） |
| | - `last_verified_at`: サーバー時刻（RevenueCatの`request_date`、UTC epoch ms） |
| | - `last_verified_uptime`: 検証時の端末uptime（`react-native-device-info`の`getUptime`、ms単位） |
| | - `entitlement_expiration`: エンタイトルメント有効期限（RevenueCatの`expirationDate`、UTC epoch ms。lifetime時はnull） |
| 対象エンタイトルメント | RevenueCatの `pro` エンタイトルメントID（設定で変更可能）を確認対象とする |
| 時刻表現 | すべてUTC epoch milliseconds（整数）で保存。比較時はms同士で直接比較 |
| 永続化タイミング | オンライン検証時（RevenueCat SDK `CustomerInfo` 取得成功時）に4項目を同時に更新 |

**オンライン検証時の処理:**

| 条件 | 処理 |
|------|------|
| `entitlement.isActive == true` | `entitlement_active = true`、`entitlement_expiration = expirationDate`（nullならlifetime）を保存 |
| `entitlement.isActive == false` または対象エンタイトルメントなし | `entitlement_active = false`、`entitlement_expiration = 0`（明示的に無効化）を保存 |
| 検証成功時 | `last_verified_at`、`last_verified_uptime` を現在値で更新 |

> **注記**: 未加入時（`isActive == false`）は `entitlement_expiration` を null ではなく 0 に設定することで、nullがlifetimeを意味するケースと明確に区別する。これにより、非加入ユーザーがオフライン判定でProを許可される実装ミスを防ぐ。

| 経過時間の算出 | `elapsed = currentUptime - last_verified_uptime`（壁時計ではなく単調増加時間で計算） |
| uptime異常検出 | `currentUptime < last_verified_uptime`（端末再起動を示す）の場合、**フォールバックなしで即座にオンライン再検証を要求** |
| 再起動時の挙動 | オンライン再検証を要求。検証成功まで無料プラン扱い（ただし、オンラインならほぼ即時復帰）。**壁時計フォールバックは再起動時には適用しない** |
| 壁時計の補助利用 | uptime正常時（`currentUptime >= last_verified_uptime`）の**追加検証**として、`currentTime >= last_verified_at`かつ`currentTime - last_verified_at <= 7日`も確認 |
| 時刻操作検出 | `currentTime < last_verified_at`（壁時計巻き戻し）検出時は即無効化してオンライン再検証を要求 |

**オフラインPro許可の最終判定式（統合）:**

以下の**すべて**を満たす場合にのみPro機能を許可:

| 条件 | 判定式 | 説明 |
|------|--------|------|
| ①エンタイトルメントアクティブ | `entitlement_active == true` | オンライン検証時にProエンタイトルメントがアクティブだった |
| ②エンタイトルメント有効期限 | `entitlement_expiration == null \|\| entitlement_expiration > currentTime` | サブスク期限がnull（lifetime）または期限内。0の場合は無効 |
| ③uptime経過時間 | `currentUptime >= last_verified_uptime && elapsed <= 7日` | 再起動なし かつ グレース期間内 |
| ④壁時計整合 | `currentTime >= last_verified_at && currentTime - last_verified_at <= 7日` | 壁時計が巻き戻されていない かつ グレース期間内 |

いずれか1つでも満たさない場合は **無料プランにフォールバック** し、オンライン再検証を要求する。

**欠損値・破損値の扱い:**

| 状態 | 判定 | 挙動 |
|------|------|------|
| 初回起動オフライン | 4値のいずれかが存在しない | Pro無効化 → オンライン検証を要求 |
| secure-store消失・破損 | 4値のいずれかが欠損・パース失敗・型不正 | Pro無効化 → オンライン検証を要求 |
| 値の不整合 | `last_verified_uptime` < 0、`last_verified_at` が未来時刻など論理的に不正 | Pro無効化 → オンライン検証を要求 |
| `entitlement_active == false` | オンライン検証時に未加入だった | Pro無効化（条件①を満たさない） |
| `entitlement_expiration == 0` | オンライン検証時に未加入だった（明示的に無効化済み） | Pro無効化（条件②を満たさない） |
| `entitlement_expiration == null` | RevenueCatが非期限エンタイトルメント（lifetime等）を返し、かつ `entitlement_active == true` | 条件②を満たす（無期限として有効） |

> **注記**: `entitlement_expiration == null` はRevenueCat SDKが `expirationDate` を null で返却した場合（lifetime subscription等）に限定される。**未加入時は `entitlement_expiration = 0` かつ `entitlement_active = false` を設定するため、nullと未加入を混同しない**。4値が揃っていない場合は常にPro無効化とし、課金保護を優先する。

> **設計意図**: 壁時計（ローカル時刻）は「未来に進めてから元に戻す」操作で回避可能なため、端末uptime（単調増加カウンタ）をグレース期間の主要判定基準とする。uptimeは端末再起動でリセットされるため、**再起動時は壁時計フォールバックを許可せず、必ずオンライン再検証を要求する**。これはセキュリティを優先する設計判断であり、通常端末再起動後にネットワーク接続がある環境が多いことを前提とする。壁時計は再起動していない場合の追加検証（uptime + 壁時計の二重チェック）として使用し、改ざん検出に利用する。**エンタイトルメント有効期限チェックを統合することで、期限切れでもグレース期間内ならProが許可される抜け穴を防ぐ**

## 3. 非機能要件

### 3.1 オフラインファースト

| 項目 | 仕様 |
|------|------|
| データ保存 | AsyncStorage（端末内ローカル保存） |
| ネットワーク依存 | 課金処理以外はオフラインで動作 |
| 同期機能 | なし（端末単体で完結） |

#### 3.1.1 データスキーマ管理

| 項目 | 方針 |
|------|------|
| スキーマバージョン | AsyncStorageに `schemaVersion` キーで保持（初期値: 1） |
| マイグレーション | アプリ起動時に現在のスキーマバージョンを確認し、必要に応じてマイグレーション関数を順次実行 |
| 後方互換性 | 新バージョンのアプリは旧スキーマのデータを読み込み、最新形式にマイグレーションする |
| マイグレーション失敗時 | エラーログを記録し、ユーザーに「データ復旧のためサポートへ連絡」を促すダイアログを表示。アプリは**読み取り専用モード**で起動継続（新規作成・編集・削除を無効化し、閲覧のみ許可）。ダイアログには「再試行」ボタンを設け、ユーザー操作でマイグレーションを再実行可能とする。**PDF出力・CSVエクスポートはPro判定を維持**（Pro限定機能は読み取り専用モードでもPro契約が必要） |
| テスト方針 | 各バージョン間のマイグレーションをユニットテストで検証。実データのスナップショットを使用した結合テストを推奨 |

> **設計意図**: AsyncStorageに保存するデータ構造は将来変更される可能性があるため、バージョン管理とマイグレーション機構を初期から用意する。これにより、アップデート時のデータ互換性問題を防ぎ、ユーザーデータを保護する

### 3.2 セキュリティ

| 項目 | 仕様 |
|------|------|
| 業務データ保存場所 | 端末内ローカル保存（アプリが能動的にクラウド送信しない。ただしOSバックアップは対象となり得る：3.2.2参照） |
| 課金データ | RevenueCat/Apple/Google へ送信（購入処理に必須） |
| 認証 | なし（端末ロックに依存） |
| 一般データストレージ | AsyncStorage（書類、単価表等） |
| 機微情報ストレージ | expo-secure-store（振込先口座、インボイス番号） |
| 端末保護 | OSの端末ロック（PIN/Face ID等）に依存 |

#### 3.2.1 データ分類と保存方針

| 分類 | 対象データ | ストレージ | 暗号化 |
|------|-----------|-----------|--------|
| 一般業務データ | 見積書、請求書、単価表、取引先名・住所、明細（品名・金額等） | AsyncStorage | なし |
| 機微情報 | 振込先口座情報、インボイス番号 | expo-secure-store | OS標準暗号化（Keychain/Keystore） |
| 機微情報スナップショット | 書類IDに紐づく発行者情報（インボイス番号・振込先口座） | expo-secure-store（`issuer_snapshot_{documentId}`） | OS標準暗号化（Keychain/Keystore） |
| 課金判定ローカルキャッシュ | `entitlement_active`、`last_verified_at`、`last_verified_uptime`、`entitlement_expiration` | expo-secure-store | OS標準暗号化（Keychain/Keystore） |
| 課金情報 | 購入履歴、サブスク状態 | RevenueCat（外部送信） | TLS通信 |

> **注記**: 機微情報はexpo-secure-storeを使用し、iOS Keychain / Android Keystoreによるハードウェアレベルの暗号化で保護します。一般業務データはAsyncStorageに保存し、端末ロックによる保護に依存します。

> **残余リスクと対策**:
> - **リスク**: 一般業務データ（取引先名・住所、請求金額等）はAsyncStorageに平文保存されるため、端末が一時的にアンロックされた状態でのアクセスやOSバックアップ経由での漏洩リスクがあります。
> - **現行対策**: 端末のPIN/生体認証ロックに依存。アプリ独自の認証機能は未実装。
> - **将来検討**: アプリロック（PIN/生体認証）機能、業務データの暗号化、バックアップからの除外設定をv1.x以降で検討。
> - **データ取扱説明画面**: 上記のリスクと対策について「データ取扱説明」画面でユーザーに明示します。

#### 3.2.2 OSバックアップ方針

| プラットフォーム | バックアップ | 方針 |
|-----------------|-------------|------|
| iOS | iCloudバックアップ | デフォルトでバックアップ対象。ユーザーがiCloud設定で除外可能 |
| Android | Googleバックアップ | デフォルトでバックアップ対象。ユーザーが設定で除外可能 |

> **注記**: アプリデータはOSのバックアップ機能によりiCloud/Googleへ送信される可能性があります。「業務データは端末内のみ」とは、**アプリが能動的にクラウドへ送信しない**ことを意味し、OSバックアップによるクラウド保存はユーザーの端末設定に依存します。データ取扱説明画面でこの旨を明示します。

## 5. ユースケース

### UC-01: 見積書を作成する

**アクター:** ユーザー
**前提条件:** アプリ起動済み（認証不要）
**基本フロー:**

1. 書類一覧画面で「+」ボタンをタップ
2. 「見積書を作成」を選択
3. 取引先名を入力
4. 明細行を追加（品名、数量、単価を入力）
5. 「保存」をタップ
6. 見積書が保存され、一覧に表示される

**代替フロー:**
- 4a. 単価表から選択して明細を追加
- 5a. 入力不備がある場合、エラーメッセージを表示

### UC-02: 見積書を請求書に変換する

**アクター:** ユーザー
**前提条件:** 見積書が作成済み
**基本フロー:**

1. 書類編集画面で「請求書へ変換」をタップ
2. 確認ダイアログで「作成」を選択（※書類番号が消費される旨を表示）
3. 請求書が自動保存・採番され、編集画面に遷移
4. 必要に応じて支払期限等を入力して再保存

### UC-03: PDFを出力して共有する【Pro】

**アクター:** Proユーザー
**前提条件:** 書類が作成済み、Proプラン加入済み
**基本フロー:**

1. 書類編集画面で「プレビュー」をタップ
2. プレビュー画面で「PDFで共有」をタップ
3. PDFが生成され、共有シートが表示
4. 送信先を選択して共有

**代替フロー:**
- 2a. 無料ユーザーの場合、Proアップグレード案内を表示

### UC-04: Proプランに加入する

**アクター:** 無料ユーザー
**前提条件:** App Store課金が有効
**基本フロー:**

1. Pro機能を利用しようとする（PDF出力等）
2. 「Proにアップグレード」を選択
3. Paywall画面で商品を確認
4. 「購入する」をタップ
5. App Storeの課金画面で決済
6. 購入完了、Pro機能が利用可能に
